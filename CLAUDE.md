# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Altunmak CRM" (`Altunmak CRM` / package name `makina-crm`) — a Windows desktop CRM for Altuntaş Makina, built with React (Vite) for the UI and Electron as the desktop shell. It tracks customers/machines, dealers, service requests, machine resale history, stock, spare parts, notes, and finance, and persists everything to a SQLite database (`data.db`) via `better-sqlite3` in `electron/db.cjs`. Supports a multi-user LAN mode: one PC acts as server (embedded Express + JWT, `electron/server.cjs`), others connect as HTTP clients; single-user local mode also works with no configuration.

## Commands

```bash
npm install          # install deps
npm run dev           # vite dev server + electron, hot reload (concurrently + wait-on)
npm run build         # vite build only (outputs to dist/)
npm run build:win     # vite build + electron-builder --win → release/*.exe (NSIS installer)
npm run preview       # vite preview of the built renderer (no electron shell)
npm run release       # build:win + electron-builder --publish always (pushes to GitHub Releases, needs GH_TOKEN env var)
```

`npm test` (vitest) runs the test suite: `tests/merge.test.js` covers the pure conflict-merge planner (`src/lib/merge.js`) under node; `tests/db-electron.test.js` spawns Electron to exercise the real SQLite layer (`scripts/tests/db-roundtrip.cjs` — migration, field roundtrip, table-skip integrity, audit retention), because better-sqlite3 is built against Electron's ABI and won't load under plain node. Pure helpers are covered by `tests/utils.test.js` / `tests/mail-templates.test.js` / `tests/permissions.test.js` / `tests/server-authz.test.js` (server-side write authorization, `electron/serverAuth.cjs`); component/regression tests live in `tests/ui/*.test.jsx` and run under jsdom via a `// @vitest-environment jsdom` pragma at the top of each file (plus explicit `afterEach(cleanup)` — auto-cleanup doesn't fire without vitest globals). `tests/server-security.test.js` spawns Electron to run the real embedded server end-to-end (`scripts/tests/server-security.cjs` — auth/401, per-user write authorization/403, admin-gating, password length, refresh-internal loopback, login rate-limit/429), same reason as the SQLite tests. **Rule: every bug fix ships with a test that would have caught it** (the UI tests exist because the permission-leak and "field not saved" bugs did). `npm run lint` (ESLint 9 flat config, `eslint.config.mjs`) targets real errors only — warnings are informational, keep the error count at zero. There is still no TypeScript.

To cut a new auto-update release: bump `"version"` in `package.json`, then run `set GH_TOKEN=ghp_...` (Windows) and `npm run release`. Auto-update (electron-updater) only works in the installed/packaged app, never in `npm run dev` or an unpacked build — `electron/main.cjs` explicitly checks `app.isPackaged` before touching the updater.

## Architecture

### Process split (Electron)
- `electron/main.cjs` — main process. Owns the data file at `app.getPath("userData")/data.json`, all `ipcMain.handle` channels (`crm:load/save/backup/restore/chooseFolder/writeBackup`, `updater:*`, `app:uninstall`, `app:printHtml`), window creation (splash + main window), single-instance lock, and security hardening (`contextIsolation: true`, `nodeIntegration: false`, blocks `will-navigate`/new windows to anything non-local).
- `electron/preload.cjs` — the only bridge between renderer and main, via `contextBridge.exposeInMainWorld`. Exposes three globals to the renderer: `window.crmStorage` (data persistence), `window.appPrint` (print-to-preview-window), `window.appUpdater` (update lifecycle events), `window.appControl` (uninstall).
- `src/main.jsx` — React root, mounts `<App />` wrapped in `<ErrorBoundary>`.
- `src/ErrorBoundary.jsx` — class component catching render errors app-wide; shows a Turkish fallback screen with a "Yeniden Başlat" button instead of a white screen.
- `src/App.jsx` — owns all top-level state and the sidebar/tab shell. There is intentionally no router; navigation is a `tab` state string with conditional rendering against a `TABS` array defined near the top of the file.

### Module layout (post-split)
The UI used to live entirely in one ~4700-line `App.jsx`. It's now split as follows — when adding code, put it in the matching module rather than back into `App.jsx`:
- `src/lib/constants.js` — pure data: embedded base64 logo reference, `ALTUNMAK_MODELS`, `COUNTRIES`/`COUNTRY_EN`/`COUNTRY_ALT`, `CITIES_TR`, seed arrays (`INIT_*`), `CURRENCIES`/`CUR_SYM`/`SALE_TYPES`/`DEFAULT_KDV_RATE`/`BACKUP_SCHEMA_VERSION`/`BACKUP_APP_TAG`/`SERVICE_TYPES`/`REPAIR_PLACES` (4 options: Yerinde Onarım, Fabrikada Onarım, Kargo, Fabrika Teslim)/`APP_VERSION`.
- `src/lib/utils.js` — pure functions: date/currency formatting (`fmtTR`, `fmt`, `fmtCur`, `parseMoney`), Turkish-aware lowercasing (`trLower`), KDV/VAT math (`calcKDV`, `extractKDV`), a module-level mutable ID counter (`nextId`/`uid()`/`bumpId()`/`getIdCounter`/`setIdCounter` — not React state, used for generating new record IDs across the whole app).
- `src/components/ui.jsx` — UI primitives: `Icon`, `Field`, `Input`, `Warn`, `Select`, `MoneyInput`, `Btn`, `StatCard`, `Modal`, `ConfirmDialog`, `Pagination`, `CountryCityFields`, plus the `EMAIL_RE`/`PHONE_RE` regexes. All styling is inline `style={{...}}` objects — there is no CSS framework, CSS module, or styled-components.
- `src/hooks/useSimpleDefList.js` — shared hook factoring out the add/inline-edit/delete logic common to `KalipManager` and `PartManager` (both simple `{ad}`-only CRUD lists). `ModelsManager` is deliberately **not** built on this hook — it's modal-based, multi-field, and has separate std-vs-custom edit modes, so forcing it into the same shape would be an awkward abstraction.
- `src/hooks/useFilteredList.js` — shared hook factoring out the repeated "search + filter + paginate" pattern used by list screens (`Customers`, `SimpleDealers`, `Stock`). Takes the raw array plus `{ searchFields, searchFn, filterFn, sortFn, perPage }`; returns `{ search, setSearch, page, setPage, filtered, paged, perPage }`. `searchFields` covers plain own-field matching; `searchFn(item, query)` is for cases where the match target is reached via a join. `Customers` only takes the search+filter+page plumbing from the hook — its groupByFirm dedupe and multi-column sort run after the hook's `filtered`, since both need to operate in a specific order relative to each other that a generic `sortFn` param can't express. `Notes.jsx` keeps its own hand-rolled pager (different page size, no shared `Pagination` component) and wasn't migrated.
- `src/components/*.jsx` — one file per feature component, each a named export: `Dashboard`, `Customers` (shared by both the "Müşteriler" and "Bayiler" tabs via props like `isCustomer`/`entity`/`title`; its `detailView` modal is also where the former standalone "Makina Geçmişi", "Servis ve Yedek Parça", and "Extra Kalıp" tabs' functionality now lives — see below), `SimpleDealers`, `ServiceForm` (servis ekle/düzenle modal body, see below), `PartSaleForm` (Extra Kalıp satışı ekle/düzenle modal body, see below), `ModelsManager`/`KalipManager`/`PartManager` (used inside `Settings`), `Notes`, `Settings` (backup/restore, CSV/XLSX export, update UI, KDV rate, model/mold/part definitions), `Stock`, `Finance`.
- `src/components/ServiceForm.jsx` — the "Yeni Servis Talebi"/edit form (customer picker, type/repairPlace, servis+parça ücreti with warranty-aware free/paid logic, değişen parçalar). It owns its own `custSearch` state internally; the caller owns `form`/`setForm` and provides `onSave(parcaUcretsizMi)` (the form computes `parcaUcretsizMi` since it needs `customers` to check warranty status, but doesn't know whether the caller is doing an add or an edit) and `onCancel`. Currently only used from `Customers.jsx`'s detail view (see below) — there used to be a standalone "Servis ve Yedek Parça" tab (`Services.jsx`) that also used it, but that tab was removed.
  - **"İşlemi Yapan Firma" = "Diğer" (anlaşmasız dış servis):** the customer's own non-contracted repair shop. Picking "Diğer" reveals firm name/contact/phone/country-city fields (`islemFirmaAd/Yetkili/Tel/Ulke/Sehir`) saved ONLY to the service record — no customer/dealer record is created. `islemFirma` holds the sentinel string `"Diğer"`; the real name is in `islemFirmaAd` (resolve for display with `islemFirmaGoster(sv)` / gate with `disServisMi(sv)` in `lib/utils.js`). Because `"Diğer"` ≠ factory name, `isAltuntasServisi` already treats it as external — service fee not counted as our revenue, parts debt attributed to the external firm (same path as anlaşmalı bayiler). PartSaleForm mirrors this with a "Satış Yapan Firma" selector (Fabrika + bayiler + "Diğer") and `satisFirma*` fields (`satisFirmaGoster`/`partSaleDisFirmaMi`). Shown in the machine-history timeline (`MachineTimeline.jsx`) and service/machine print reports. New DB columns are plain TEXT (auto-read via `SELECT *` + `...rest`; INSERT + SCHEMA + ensureColumns still required — see `SERVICES_DIS_FIRMA_COLUMNS`/`PART_SALES_SATIS_FIRMA_COLUMNS` in `db.cjs`).
- `src/components/PartSaleForm.jsx` — the Extra Kalıp satışı/çıkışı add/edit form, same pattern as `ServiceForm` but simpler (no warranty-aware free/paid branch — a mold sale is always paid, so `onSave` takes no extra computed argument). Also currently only used from `Customers.jsx`'s detail view — there used to be a standalone "Extra Kalıp" tab (`Parts.jsx`) that also used it, but that tab was removed too.

### Customers' detail view absorbed three former tabs: "Makina Geçmişi", "Servis ve Yedek Parça", "Extra Kalıp"
All three used to be separate top-level tabs. They were removed one at a time and merged into `Customers.jsx`'s `detailView` modal (opened by clicking a customer/machine row), on the reasoning that once every action is reachable from a customer's own profile, maintaining parallel "all customers" list tabs for the same data is redundant for this app's actual usage. **This was an explicit, deliberate product decision by the user** (not just a refactor convenience) — they were told the trade-off up front (losing cross-customer search like "all services by technician X", and losing a flat at-a-glance "who owes how much" list) and chose to proceed anyway, wanting to see how it feels in practice. If a future request asks to bring back a standalone list tab, that's not a contradiction of earlier work — it's revisiting a knowingly-provisional choice. Key points if touching this:
- `detailViewId` (not the record itself) is the state — `detailView` is derived live via `customers.find(c => c.id === detailViewId)` every render, so it reflects mutations (edits, 2nd-hand transfers, service/kalıp ödeme toggles) immediately instead of going stale like a stored snapshot would.
- The modal is two-column when the firm has more than one machine: a left "Bu Firmanın Makinaları" sidebar (click a row to switch which machine's detail is shown) and a right column with the field grid, "Sahiplik Geçmişi" (ownership chain, if resold), and the combined "Makina Geçmişi" timeline (sale + services + Extra Kalıp sales + warranty-end, built from `services` and `partSales`). With only one machine, the sidebar is hidden and the right column takes the full width.
- There is only ONE edit form for a customer/machine record — `openEdit`/`form`/`save` (the rich one with finance fields, `kaliplar` array, stock-linked serial selection). The old standalone `MachineHistory` tab had a second, simpler, partially-incompatible edit form (different field names: `kalip` singular vs `kaliplar` array, `note` vs `aciklama`) — that was deliberately dropped rather than carried over, to avoid two editors diverging on the same record. The same one-editor principle applies to services (`ServiceForm`) and Extra Kalıp sales (`PartSaleForm`): each has exactly one shared form, opened either fresh (pre-filled with `customerId: detailView.id`) or in edit mode from the timeline, never a second parallel form.
- "Yeni Sahip" (2nd-hand ownership transfer: `saveNewOwner`) and "Yazdır" (`printMachineReport`, the printable "Makina Servis ve Yedek Parça Geçmişi Raporu" for the whole machine) live in `Customers.jsx`, rebased onto `detailView`/`detailHistory`/`detailTimelineEvents`. There's also `printServiceForm(sv)` — the single-service "Servis Formu" printout (ported from the old `Services.jsx`, which had its own per-row Yazdır button); its button sits inline next to that service's title in the timeline, not in the modal's bottom action row, since it's per-record rather than per-machine.
- The "Satış Yapan" field (both in the main edit form and in the "Yeni Sahip" form's "Satan Firma" field) is a quick-pick `<Select>` (Fabrika / Bayiler / Önceki Sahip or Mevcut Sahip) layered over a free-text `<Input>` that holds the actual value — picking an option just fills the input, so any name (e.g. a previous owner who sold privately, not through the factory or a dealer) can also be typed directly.
- The timeline's service AND Extra Kalıp ("Kalıp Verildi") entries are actionable, not just read-only: the "Makina Geçmişi" box header has "Yeni Servis Talebi" and "Extra Kalıp Satışı" buttons (open `ServiceForm`/`PartSaleForm` pre-filled with that customer); each timeline entry's title is clickable to reopen the matching form in edit mode for that record; and the servis/parça/kalıp ücreti lines each have a one-click "Ödendi"/"Ödenmedi" toggle (`toggleServisOdendi`/`toggleParcaOdendi`/`togglePartSaleOdendi`) that flips the flag directly without opening a form.
- `Customers` needs `partSales`/`setPartSales` and `parts` as props (timeline/print report/Extra Kalıp form, and the "Değişen Parçalar" picker in the embedded `ServiceForm`); `App.jsx`'s `customers` tab render call passes all of them.
- Debt detection is unified across all four sources (customer's own `kalanBorc`, unpaid service fee, unpaid parça fee, unpaid Extra Kalıp sale) via `customerHasAnyDebt(customer, services, partSales)` in `lib/utils.js` (built on `isServisBorcluMu`/`isPartSaleBorcluMu`, also shared). `Customers.jsx`'s "₺ Borçlu Firmalar" filter pill and its row highlight (`hasDebt`, yellow background) both use it — a customer can show up there purely from an unpaid service or mold sale, with no `kalanBorc` at all. `Dashboard.jsx`'s borçlu-firma count and its "Borçlu Firmalar" modal use the same three predicates, and the modal's single "Müşterilerde Görüntüle" button now routes to `Customers` with the debt filter (there's nowhere else left to route to, since the Services/Parts tabs are gone).
- `App.jsx`'s `TABS` array no longer has `services`/`parts`/`machines` entries; `Dashboard.jsx` no longer takes `onGoServices`/`onGoParts` props (removed along with the tabs they pointed to). The "Servis Kayıtları" stat card on the Dashboard is now a plain non-clickable count (no tab to navigate to).
- `src/App.jsx` (last in the chain): owns all top-level state (`customers`, `dealers`, `services`, `stock`, `notes`, `parts`, `partSales`, `kalipDefs`, `standardModels`/`customModels`, `factory`, `appSettings`, geo data, toast, current `tab`) and renders the sidebar + the active tab's component, passing data/setters down as props (no Context, no Redux/Zustand — plain prop drilling).

### Data persistence pattern
- On mount, `App` calls `window.crmStorage.load()`. In local mode this reads from SQLite via `electron/db.cjs`; in client mode it fetches from the server via `electron/ipc/data.cjs`. If Electron APIs are absent (e.g. `npm run preview`), storage silently no-ops and data only lives in memory.
- Any change to the tracked state arrays triggers a `useRef`-backed 500ms debounced `window.crmStorage.save(...)` writing the entire state blob at once. **The blob is the wire/IPC format only — SQLite does NOT store it as a blob.** `db.cjs` unpacks it into normalized, column-per-field tables (`customers`, `services`, `dealers`, …); the `meta` table holds only small key/value scalars (`dataVersion` etc.). Consequence: **adding a field to a record is not enough to persist it.** A new column must be added in four places or the value is silently dropped on the next load — `SCHEMA_SQL`, an `ensureColumns` entry in `applyColumnMigrations`, the `INSERT` column list + params, and the read/SELECT mapping. This has bitten before (`factory.ilce`); a roundtrip test in `scripts/tests/db-roundtrip.cjs` is the guard.
- SQLite writes are atomic via WAL mode; `electron/db.cjs` also keeps the old `data.json` atomic-write pattern (`fs.renameSync`) for the JSON fallback path.
- Schema migrations: `SCHEMA_SQL` holds the `CREATE TABLE` baseline, but newer columns are added incrementally via `ensureColumns` (idempotent). `migrateFromJsonIfNeeded` has three init branches — existing DB, clean install, and JSON→SQLite import — and **all three must call `applyColumnMigrations(conn)` after `SCHEMA_SQL`** (that function bundles every `ensureColumns` call). Regression to avoid: clean install used to run `SCHEMA_SQL` only, so a brand-new machine's first session was missing later columns (e.g. `uretim_formlari.baslangicTarihi`) and writing such a record crashed until the second launch. When you add a new column, add it to `applyColumnMigrations` (not just one branch); `scripts/tests/db-clean-install.cjs` guards the clean-install path.
- Multi-user mode: server PC runs `electron/server.cjs` (Express + JWT). Client PCs poll `/api/version` every 30 s and listen for `server:dataChanged` push events. Optimistic locking with `dataVersion` prevents silent overwrites (409 on conflict → reload). Entity-level pessimistic locking via `useLock` hook + `/api/lock` endpoint (2-min TTL, heartbeat every 60 s, fail-open).
- Permissions are enforced on the server too, not just in the UI. The `makeCanDo`/`READONLY_SERVER_PERMISSIONS` model in `src/lib/permissions.js` hides buttons client-side; `POST /api/data` additionally re-checks it (`electron/serverAuth.cjs`): it diffs the incoming blob against the stored one section-by-section (`degisenBolumler`, key-order-stable) and rejects (403) any changed section the user's permission group blocks — so a crafted HTTP request from a read-only/restricted client can't overwrite data it can't edit in the UI. `BLOB_SECTIONS` lives in `serverAuth.cjs` (single source; `db.cjs` imports it) and every section must be mapped in `SECTION_GROUP` (a test enforces coverage). Limitation by design: the whole-blob save model can't distinguish add/edit/delete, so a group that is non-empty-but-partial (e.g. edit-but-not-delete) is trusted at the section level. Full/admin users skip the diff (`kisitliMi` gate). This matters most once the server is reachable beyond the LAN (e.g. over a Tailscale VPN).
- Two server-authoritative logs, both admin-only, both pruned after 12 months: **security log** (`security_log`, "Kullanıcı Geçmişi", via `logSecurity` in `server.cjs`) and **audit log** (`audit_log`, "İşlem Geçmişi", via `db.writeAuditEntry`). Security log records auth/user-management events plus (added later): `sifre_degistirildi` (`PATCH /auth/me/password`), `cikis` (`POST /auth/logout`, called by the client's `server:logout` IPC before dropping the token), `oturum_yenilendi` (`GET /auth/me`, ~daily), `2fa_kurulum_baslatildi` (`POST /auth/2fa/setup`), `kilit_devralindi`/`kilit_birakildi` (force-steal / release on the lock endpoints). Server-PC-only local admin ops are logged straight to `security_log` from the IPC layer (not via HTTP) through `logYerelGuvenlik` (`ipc/data.cjs`) / `kaydetYerelGuvenlik` (`ipc/system.cjs`) — actor = config username, ip = `"yerel"`, no-op when the local DB is inactive (client PCs): `ilk_admin_olusturuldu`, `sunucu_baslatildi`/`sunucu_durduruldu`, `tls_modu_degistirildi`, `sertifika_yenilendi`, `sunucu_ayari_silindi`, `yedek_alindi`, `yedek_parolasi_degistirildi`, `uygulama_kilidi_kuruldu`/`_kaldirildi`/`_parolasi_degistirildi`.
- The audit log is otherwise **client-driven** — the client POSTs `/api/audit` with self-declared `action/entity` per user action (`src/lib/audit.js`, 61 call sites). To close the "crafted HTTP write leaves no trace" gap, `POST /api/data` now writes a server-authoritative **`veri_kaydedildi`** row for **every** write (admin included), entity `"sunucu"`, `entity_name` = changed section labels from `degisenBolumler` via `BOLUM_ADLARI`. This reads the stored blob on every write (the admin perf-skip of the authz diff no longer applies). File ops audit server-side too: `POST /api/files/upload` → `yuklendi`/`dosya`, `DELETE /api/files/:name` → `silindi`/`dosya`. Any new action/entity string must be added to the label maps in `SettingsSecurityLog.jsx`/`SettingsAuditLog.jsx` or it renders as its raw key; `SettingsSecurityLog` also surfaces the failed-login reason (`detail.sebep`) inline next to the `giris_basarisiz` label, and `SettingsAuditLog` now shows a Rol column. Covered end-to-end in `scripts/tests/server-security.cjs`. **Harness gotcha:** under Electron the check script's `process.exit(1)` is deferred (execution continues), so the failure path `return`s before printing the `TUM KONTROLLER GECTI` sentinel — the vitest wrapper asserts both exit 0 AND that sentinel, so a failed check can't slip through as a pass.
- Manual backup/restore and scheduled auto-backup go through separate IPC channels (`crm:backup`, `crm:restore`, `crm:chooseFolder`, `crm:writeBackup`) and write timestamped JSON files.
- New records get IDs from the global `uid()` counter, not from `crypto.randomUUID()` or DB-assigned IDs — when loading saved data, `bumpId(...)` scans existing records to advance the counter past any existing max ID.

### Faaliyet Haritası (the map tab)
- `src/components/Harita.jsx` + `src/components/harita/HaritaSvg.jsx` + `src/lib/mapStats.js`. Three drill levels: Dünya → ülke/il → ilçe. It's an "activity" map: every place that has sales is filled a **single logo orange** (`--hSatis`, not a density ramp) and gets a neutral location pin (`tur:"satis"`, theme-aware `--n900` so it stands out from the orange fill and the blue/green dealer pins) with the place name as a label; the factory and dealers/services are the other pins. **Every drawn place is name-labeled**, selling or not: selling ones via their satis pin, non-selling ones via a muted standalone label at the shape centroid (`yolMerkezi(d)` in `mapStats.js`). Labels live in the pin layer so they stay screen-px sized (readable when zoomed in, overlapping when zoomed out — accepted).
- **Map geometry is generated at build time, not fetched at runtime.** `scripts/gen-map-paths.cjs` turns upstream boundary data into plain JS modules under `src/lib/map/` (~8MB, **committed on purpose** — the app must work offline and the generator needs network + a local cache). Don't hand-edit anything in `src/lib/map/`; change the generator and re-run it. Companion: `scripts/gen-arkaplan-adlar.cjs` → `src/lib/map/world-arkaplan.js` (`ARKA_PLAN_AD` = Turkish names + projected centroids for the ~92 world countries **outside** the 86-country sales list, which `world.js`'s `ARKA_PLAN` stores nameless; territories below a projected-area threshold are skipped to avoid clutter, except a `KUCUK_ULKE` allowlist of small sovereign states (Luxembourg, Malta, Monaco, Andorra, Vatican, small island nations…) which are labeled regardless of size; dependent territories and geographic features stay excluded). It's **offline** — only `world-atlas` (local) + `d3-geo` (devDep) — and must reproduce `gen-map-paths.cjs`'s exact world projection + used-country set or the labels drift off the paths.
- Those modules are lazy-loaded via `import.meta.glob`, so Vite code-splits them and the main bundle only grows ~43KB. `d3-geo`/`topojson-*` are **devDependencies only** (build-time); never import them from renderer code.
- **Licensing matters here.** Sources: Natural Earth (public domain), GeoNames (CC BY), and geoBoundaries TUR ADM1+ADM2 (**ODbL 1.0 — share-alike**, pinned to a specific commit). Attribution is surfaced in-app under Ayarlar > Uygulama > Harita Veri Kaynakları. If you swap or add a source, update that accordion and ATTRIBUTION.md.
- Gotchas already paid for: pin scale must be normalized to screen px (`sigdirmaOlcegi()`) or pins shrink on tall viewBoxes; polygon winding is fixed by measuring area > 2π and reversing; districts come from geoBoundaries ADM1 (not Natural Earth provinces, which misfile Tuzla under Kocaeli).

### Servis Panosu (the "servis" tab / service-floor kiosk)
- `src/components/ServisPanosu.jsx` — a Kanban board of services by work status **`durum`** (a new plain-TEXT service field: "Bekliyor" / "Yapılıyor" / "Tamamlandı"; **only services that have a `durum` appear** — legacy/durum-less services stay off the board and are managed from the customer detail). Native HTML5 drag-drop (no library) moves a card between columns → `setServices(p => p.map(s => s.id===id ? {...s, durum} : s))`. Each card has a technician `<select>`, and "Yeni Servis Talebi"/card-edit open **the exact same full `ServiceForm` as the customer detail** (explicit user requirement — the board form must be identical to the customer service-request form, parts/money/stock included; a new service defaults to `durum:"Bekliyor"`). Its save replicates `CustomerDetailModal.saveService` (`rec = {...form, customerId, parcaUcretsizMi, parcaUcreti, parcaUcretiAltuntastan, parcaCurrency}`). **Because the form is identical, the parts stock-deduction is too:** the deduct/restore logic is factored into `src/lib/servisStok.js` (`servisParcaDus`/`servisParcaGeriAl`, writing `partStock`/`partStockLog`), shared by both the board and `CustomerDetailModal` so they never drift. The board therefore needs `parts`/`dealers`/`kdvRates`/`geoData`/`loadingGeo`/`setPartStock`/`setPartStockLog` props from `App.jsx`. **The identical form also includes the "Servis Resimleri / Dosyaları" upload** — the board passes `dosyalar`/`setDosyalar`/`dosyaCevrimdisi` and replicates `CustomerDetailModal.bindServisDosyalari` (binds `dosyaTaslaklari` from `onSave`'s 2nd arg to the saved service as `refType:"servis"`); the file section is gated by `canDo("cust_dosya_add")` exactly like the customer form, so it auto-hides for a user without that permission.
- **"Panodan Kaldır" (archive) on completed cards:** a Tamamlandı card has a "🗄 Kaldır" button that sets a new `panoGizli` boolean (INTEGER 0/1 in `db.cjs`, so it follows the four-place rule *with* explicit `toInt`/`toBool` — a boolean can't ride the `...rest` auto-read). This **hides the card from the board without deleting the service** — the record stays in customer detail / machine history / reports, and `durum` stays `"Tamamlandı"` so the history badge is intact. Archived cards are reachable via a collapsible "🗄 Arşivlenenler (N)" reveal under the Tamamlandı column, each with a "↩ Panoya Geri Al" (`panoGizli:false`). Dragging an archived-then-restored card, or moving any card out of Tamamlandı, clears `panoGizli` (so it can't stay hidden in a non-completed column). The board filters `panoGizli` out of its columns; setting it is a plain `services` edit, so no serverAuth change is needed (already mapped to `"servis"` via the services section).
- **Technician is a dropdown of company employees**, not free text. Employees are a small `{id, ad}` list `calisanlar`, stored via the **meta-JSON pattern** (like `partTypeDefs`; `db.cjs` write/read near partTypeDefs, no column migration), managed in Ayarlar > Firma > Firma Çalışanları (`CalisanManager.jsx`, built on `useSimpleDefList`). `ServiceForm`'s "Teknisyen" field is also this select now (needs the `calisanlar` prop, drilled App→Customers→CustomerDetailModal→ServiceForm). `services[].tech` still stores the plain name, so `CalisanManager` renames propagate via `onRename`.
- **Full kiosk** for the service-floor PC: it's a server **client** logged in as a restricted `user` with `permissions.tabs:["servis"]` (+ `customerActions:["cust_service_add","cust_service_edit"]`). When a client's only visible tab is `servis` (`kioskMode` in `App.jsx`), the whole shell (sidebar, search, tab nav) is skipped and only the board renders fullscreen. **Critical server coupling:** `electron/serverAuth.cjs` `BOLUM_SEKMELERI.services` MUST include `"servis"` (else the kiosk's status writes 403 via `sekmeEngelli`); **and because the board's service form deducts parts and can attach files, `BOLUM_SEKMELERI.partStock`/`partStockLog`/`dosyalar` also include `"servis"`** (else a parts-carrying or file-carrying service save 403s for the kiosk user). Attaching files on the kiosk additionally needs `cust_dosya_add` in the user's `customerActions` (the section otherwise stays hidden, so no 403 in practice). The kiosk user's `stockActions` must stay unset/undefined (its default) — that's the group gate; UserManager only writes a `stockActions` array if the admin explicitly restricts it, so a plain `tabs:["servis"]` + `customerActions` user passes. `calisanlar` is a new BLOB_SECTIONS entry mapped to `settings` in SECTION_GROUP/BOLUM_SEKMELERI; and `servis` is in `serverPermissionDefs.js` `ALL_TABS` so UserManager shows the checkbox. Adding `durum` follows the usual four-place DB rule (SELECT auto for plain TEXT); `panoGizli` is INTEGER so its SELECT mapping is explicit (`toBool`).

### Printing and export
- Printable documents (machine history report, service form) are built as raw HTML strings via template literals (not JSX) inside the relevant component, then sent through `window.appPrint.printHtml(html)`, which opens a visible Electron `BrowserWindow` preview with a Print/Close toolbar injected into the page (see `app:printHtml` handler in `main.cjs`). There's a browser-mode fallback (open as blob / download `.html`) when `window.appPrint` isn't available.
- **Print translations are per-document (`appSettings.translations`).** Teklif and Proforma are now edited independently (`SettingsTranslations.jsx` has NO shared "Ortak" section — the shared groups are re-emitted under each doc via `ortakGruplar(ns)`). Storage shape: `{ teklif:{TR,EN}, proforma:{TR,EN}, servis, makina, sandik, fatura }`. **The merge order in `buildPrintHtml` is load-bearing and must stay `DEFAULT < flat translations.TR/.EN (legacy shared pool) < translations[ns]` (`ceviriNs = isProforma ? "proforma" : "teklif"`)** — dropping the flat layer silently reverts existing users' customizations. `servis/makina/sandik/fatura` were already namespaced and are unaffected. `ceviri-kapsam.test.js` guards servis/makina key coverage; `print-templates.test.js` guards the teklif/proforma namespace + legacy fallback.
- **Yurt Dışı Fatura packaging note** lives in its own `paketlemeNot` form-config section (split from `paketleme` in `SettingsDocuments.jsx`; `gocFaturaNot` migrates old hidden/order/label state on mount). `buildFaturaHtml` hides the note when EITHER `hiddenFields.paketlemeNot` OR (legacy) `hiddenFields.paketleme` contains `"not"`, so un-migrated configs still hide it.
- **UserManager** surfaces Servis Panosu permissions in their own "Servis Panosu işlemleri" accordion, but they are still the `customerActions` `cust_service_*` ids (no separate server permission dimension) — the accordion shares the `customerActions` customize toggle with "Müşteri işlemleri". The split filters `CUSTOMER_ACTION_GROUPS` by the group name `"Makina Geçmişi — Servisler"` (guarded by `server-permission-defs.test.js`). Settings left-menu groups (`SETTINGS_GROUPS`) were reorganized gen-crm-style: Sunucu is its own group (out of Güvenlik), plus Uygulama/Firma/Katalog/Evrak & Süreçler/Veri Yönetimi.
- CSV export is done manually (BOM + `;`-delimited rows) via `downloadCSV()`-style helpers in `Settings`.
- XLSX export/import uses the `xlsx` (SheetJS) package directly in `Settings` for full-data import/export templates.

### Conventions to preserve when editing
- All user-facing strings and form labels are in Turkish — keep new UI text consistent with this.
- Money fields are stored as raw numbers in state and only formatted for display (`fmtCur`/`fmt`); free-text money inputs go through `parseMoney()` to normalize Turkish thousands/decimal separators (`.`/`,`).
- `normalizeSaleType()` maps legacy sale-type strings to the current four-way `SALE_TYPES` (full Faturalı/Faturasız × Yurtiçi/Yurtdışı matrix: "Faturalı Yurtiçi" / "Faturalı Yurtdışı" / "Faturasız Yurtiçi" / "Faturasız Yurtdışı"). It recognizes already-current values as-is, then falls back to substring matching for legacy data — the old three-way system ("Faturalı Yurt İçi" / "Faturalı İhracat" / plain "Faturasız" with no yurtiçi/yurtdışı qualifier) and very old "Faturalı"/"Faturasız" data. Legacy "Faturasız" (no qualifier) defaults to "Faturasız Yurtiçi" since that data never tracked the distinction. Don't assume sale type fields are always already normalized when reading old/imported data — always run them through `normalizeSaleType()` first.
- When adding a new persisted state array in `App`, remember to (1) seed it in the initial load fallback, (2) include it in the debounced save effect's dependency array and payload, (3) include it in backup export/import and the XLSX template in `Settings` if it should be portable, and (4) **if it's an id-keyed record list, add it to `MERGE_KEYS` in `src/lib/merge.js` AND add an `apply(setX, "x")` line in `App.jsx`'s `mergeLocalIntoReloaded`.** Skipping (4) silently drops that array's just-added records whenever the conflict-merge runs — on a **single server PC** this fires from the app's own 5 s version poll (`calisanlar` disappeared "after a while" exactly this way). `mergeLocalIntoReloaded` only re-applies MERGE_KEYS *adds*; edits to pre-existing records still lose to the reloaded copy, so the save path also syncs `dataVersionRef` right after a successful `save()` to stop our own write from looking like an external change and triggering that reload.
